1. Diagnóstico del benchmark actual
El benchmark vigente sirve como herramienta visual rápida, pero no como evidencia 
reproducible. Hoy mezcla descarga en vivo, definición implícita del corte temporal, identidad 
incompleta del modelo y persistencia insuficiente. El resultado es útil para inspección, pero débil 
para comparar corridas, demostrar estabilidad o reconstruir exactamente una ejecución pasada.
• La corrida depende del tiempo actual cuando no se fija explícitamente una fecha de ejecución.
• Los datos de mercado no se materializan como snapshot inmutable antes del cálculo de 
métricas.
• No se persiste un manifiesto completo de la corrida con checksums, configuración, perfil 
activo, split efectivo y commit de código.
• La interfaz mezcla benchmark rápido y benchmark serio, lo que induce a interpretar como 
reproducible algo que no lo es.
• El histórico persistido es demasiado pobre para reconstruir o auditar una corrida de forma 
fuerte.
2. Principios arquitectónicos del rediseño
• Aislamiento: el benchmark nuevo debe vivir en un módulo independiente, por ejemplo 
`again_benchmark/`, sin acoplarse a internals frágiles del pipeline TFT.
• Reproducibilidad fuerte: toda corrida oficial debe poder rehacerse a partir de un manifiesto y 
un snapshot congelado, sin depender de `now()` ni de descargas vivas.
• Gradualidad: el benchmark rápido actual puede seguir existiendo, pero claramente etiquetado 
como modo exploratorio y no certificable.
• Trazabilidad: cada corrida oficial debe registrar identidad exacta de datos, modelo, 
normalizadores, configuración, universo, split y código.
• No intrusión: la app Streamlit y el pipeline de inferencia deben consumir el módulo 
benchmark a través de una frontera pequeña; el benchmark no debe gobernar el núcleo TFT.
• Pruebas con valor: la nueva batería de tests debe demostrar determinismo, no regresión, 
coherencia temporal, persistencia correcta y calidad de los datos mostrados en la interfaz.

Fase 0 — Definición y contención
• Crear una definición formal del benchmark y fijar que el benchmark oficial nunca depende de 
descarga viva durante el rerun.
• Separar explícitamente dos modos: `live_benchmark` para exploración y `frozen_benchmark` 
para evidencia reproducible.
• Definir las rutas nuevas de artefactos, por ejemplo `benchmarks/definitions/`, 
`benchmarks/snapshots/`, `benchmarks/runs/` y `benchmarks/catalog.sqlite`.
• Decidir la política temporal única: toda corrida oficial debe fijar `as_of_timestamp`, `split_date` 
o un conjunto de anchors de evaluación.
• Prohibir desde el diseño que el benchmark oficial reescriba o reconfigure artefactos del 
pipeline TFT.
Fase 1 — Contratos, manifiestos y definiciones
• Introducir contratos explícitos como `BenchmarkDefinition`, `BenchmarkSnapshotManifest`, 
`BenchmarkRunManifest`, `BenchmarkTickerResult`, `BenchmarkSummary`, 
`BenchmarkComparisonResult`.
• Fijar un `benchmark_version` y un `definition_id` para soportar evolución controlada del 
sistema.
• Persistir en manifiesto al menos: id de corrida, timestamps, tickers, horizonte, política de split, 
fingerprint de config, checksums de modelo y normalizadores, fingerprint de snapshot y 
commit de código.
• Definir tipos de error propios: configuración inválida, snapshot incompatible, rerun no 
reproducible, definición corrupta, comparación inválida.
• Construir validadores fuertes para definición, universos, fechas, horizonte, duplicados, tickers 
faltantes y compatibilidad de artefactos.

Fase 2 — Snapshot de datos y storage
• Antes de calcular métricas oficiales, descargar y normalizar datos una sola vez, guardarlos 
como snapshot inmutable y calcular su hash SHA-256.
• Guardar el snapshot en formato tabular robusto, idealmente Parquet, más un manifiesto JSON 
adyacente.
• Separar claramente catálogo e información pesada: SQLite como índice y parquet/json para 
artefactos grandes.
• Permitir rerun exacto desde un `run_id` o desde un manifiesto sin volver a descargar datos.
• Guardar también los datos preparados para interfaz, para que la visualización histórica no 
dependa de recomputar en caliente.
Fase 3 — Runner reproducible y métricas
• Crear un runner que solo acepte definiciones explícitas y, para modo frozen, snapshots 
materializados.
• Introducir política clara de métricas por ticker y métricas agregadas globales, con semántica 
documentada y estable.
• Permitir benchmark de un solo corte congelado y benchmark rolling-origin con anchors fijos; 
ambos deben compartir contratos y storage.
• Separar cálculo de predicción, cálculo de métricas y empaquetado de resultados para evitar 
funciones monolíticas.
• Registrar siempre el universo efectivo evaluado, los descartes y las razones de descarte.
Fase 4 — Interfaz y experiencia de usuario
• Crear una sección de Streamlit específica para benchmark reproducible, desacoplada 
mediante un `ui_adapter.py` que exponga DTOs ya listos para la vista.
• Mostrar badges inequívocos: `Rápido / no congelado` frente a `Oficial / congelado / 
reproducible`.
• Ofrecer tabla comparativa de corridas, gráfico por ticker, comparación entre runs, panel de 
manifiesto y auditoría de artefactos.
• Permitir filtrar por benchmark id, run id, fecha, modelo, universo, tipo de corrida y estado de 
validación.
• Preparar datos de visualización previamente en el módulo benchmark para que la interfaz 
solo pinte y no recalcule lógica compleja.

Fase 5 — Testing, CI y certificación
• Crear fixtures congeladas de datos, manifests y outputs esperados.
• Probar rerun exacto, compatibilidad de snapshots, integridad de hashes, invariantes 
temporales, estabilidad de serialización y no intrusión sobre el núcleo TFT.
• Introducir golden tests para resultados de métricas y para payloads de visualización de 
interfaz.
• Ejecutar la nueva suite con `pytest` y dejar CI alineada con los tests reales.
• Exigir como criterio de aceptación que una corrida oficial produzca el mismo `run_id`, el 
mismo manifiesto lógico y las mismas métricas cuando se relanza con el mismo snapshot y 
los mismos artefactos.

El catálogo SQLite solo debe indexar corridas y facilitar búsqueda. Los datos pesados y los 
artefactos auditables deben vivir como archivos inmutables en disco.
7. Integración mínima con el núcleo TFT
• El módulo benchmark no debe importar componentes internos del pipeline salvo a través de 
un adaptador acotado, por ejemplo `again_benchmark.adapters.again_inference`.
• El adaptador de inferencia debe exponer una interfaz pequeña: cargar artefactos, obtener 
predicciones congelables y devolver metadata de artefactos usados.
• La app Streamlit no debe conocer el almacenamiento interno del benchmark; solo debe 
invocar un servicio/adaptador de alto nivel.
• Si el módulo benchmark falla, la pantalla de predicción TFT debe seguir funcionando porque 
no comparte estado crítico ni rutas de ejecución obligatorias.
8. Interfaz objetivo
La interfaz debe ser impecable, pero la lógica de benchmark no debe vivir en la UI.
• Panel de lanzamiento con selector de definición y modo (`rápido` / `oficial`).
• Resumen superior con run id, modelo, snapshot, split, universo y estado de validación.
• Tabla comparativa de runs con filtros y ordenación robusta.
• Gráfico principal por ticker con serie real, serie prevista y marcadores de corte/anchor.

9. Estrategia de testing
La suite nueva debe demostrar no solo que el código corre, sino que el sistema cumple su 
objetivo.
• Unit tests de contratos, validadores, serialización, hashing y manifests.
• Integration tests de `definition -> snapshot -> run -> persist -> rerun`.
• Golden tests de métricas usando fixtures congeladas.
• Regression tests de payloads preparados para interfaz.
• Tests de determinismo: misma definición + mismo snapshot + mismos artefactos -> mismo 
resultado.
• Tests de no intrusión: importar y usar `again_benchmark` no debe modificar configuración, 
artefactos ni estado del pipeline TFT.
• Tests de migración si se mantiene compatibilidad con el benchmark histórico actual.
10. Criterios de aceptación
1. Existe un módulo nuevo independiente y autocontenido.
2. El benchmark oficial puede ejecutarse y reejecutarse sin volver a descargar datos.
3. Cada corrida oficial persiste snapshot, manifiesto, métricas y payloads de interfaz.
4. Una rerun idéntica reproduce resultados y trazabilidad esperada.
5. La UI diferencia nítidamente exploración live y benchmark congelado.
6. La CI ejecuta toda la suite relevante y falla ante cualquier divergencia reproducible.
7. El núcleo TFT sigue intacto y operativamente desacoplado.
11. Secuencia recomendada de implementación
1. Crear `again_benchmark/` con contratos, errores, validación y manifests, sin tocar la app.
2. Implementar snapshots y storage en paralelo, con tests de serialización y hashing.

12. Qué no debe hacerse
• No reusar el benchmark actual añadiendo parches encima hasta convertirlo en un bloque 
opaco.
• No dejar que la UI siga decidiendo implícitamente qué es una corrida oficial.
• No guardar solo JSONs agregados sin snapshot ni manifiesto.
• No acoplar el benchmark a rutas internas del TFT que puedan cambiar con refactors futuros.
• No mezclar en una sola función descarga, inferencia, métricas, persistencia y visualización.

Apéndice B — Resultado esperado de la migración
Al final del proyecto, AGAIN tendrá dos capacidades distintas y bien separadas: predicción TFT 
como núcleo del sistema, y benchmark reproducible como módulo periférico pero profesional. 
La primera seguirá siendo el corazón operativo. La segunda servirá para evaluar, comparar y 
auditar sin poner en riesgo la estabilidad del núcleo.
